# 真实浏览器联机回归踩坑记录

这份文档记录本项目在 Codex + `agent-browser` / `chrome-devtools MCP` + 真实 Chrome 联机回归时踩过的坑、稳定做法和后续执行规范。

## 目标

- 避免重复走弯路
- 避免因为 `--cdp` 端口递增而反复申请相似权限
- 固化一套可重复执行的真实浏览器回归流程

## 已验证的稳定方案

### 0. 如果目标是“更少的同意操作”，优先用 `chrome-devtools MCP`

实践结果：

- 当前会话里，`chrome-devtools MCP` 一旦配置完成并重启生效，后续浏览器操作不再依赖新的 shell 命令
- 在 `server` 和 `client` 已经由真实环境启动好的前提下，打开页面、切页、点击、填表、查看 console/network 基本不需要新的同意
- 这比 `agent-browser` 的 CLI 流程更省审批

结论：

- 想要最低审批成本时，优先用 `chrome-devtools MCP`
- 想要继续复用固定 shell 脚本工作流时，用 `agent-browser`

### 0.2 `chrome-devtools MCP` 做多玩家联机时，不能只开普通标签页

已验证现象：

- 同一个浏览器上下文里的多个标签页会共享 `localStorage`
- 本项目用 `deviceId` 识别玩家身份
- 所以如果只开第二个普通标签页，服务端会把它识别成同一个玩家，不会形成真实双人联机

稳定做法：

- 第一个页面用默认上下文创建房间
- 第二个页面必须使用 `isolatedContext`
- 示例：`isolatedContext = player2`

结论：

- `chrome-devtools MCP` 里模拟多玩家，核心不是“多页”，而是“多隔离上下文”

### 0.2.1 不要复用旧的 `isolatedContext` 做新场景

已验证现象：

- 同一个 `isolatedContext` 会保留 `localStorage`
- 本项目的设备身份保存在 `localStorage['texas-holdem-device-id']`
- 如果复用旧上下文继续做新房间或新场景，旧房间状态、旧筹码、旧身份都会污染当前回归

稳定做法：

- 每组回归都使用全新的上下文名
- 示例：
  - `host-mcp-1`
  - `guest-mcp-1`
  - `host-mcp-3`
  - `p2-mcp-3`
  - `p3-mcp-3`

建议：

- 一个场景结束后，直接关闭相关页面
- 下一组场景重新创建新的 `isolatedContext`

### 0.3.1 `chrome-devtools MCP` 已验证通过的更高价值链路

- 两人完整一手牌：
  - `all-in`
  - `call`
  - 自动补公共牌
  - 摊牌
  - 筹码结算
- 三人补码：
  - `p2` 从 `1000` 补到 `2000`
  - `p3` 从 `1000` 补到 `4000`
- 补码边界：
  - 开局前补码会立即同步到前端和 `/api/debug/rooms/:roomId`
  - 进行中的手牌里，补码入口会隐藏
  - 玩家弃牌进入结果态后，补码入口会重新出现
- 超时自动动作：
  - 有注可跟时，超时会自动弃牌，并直接推进到下一手
  - 可 `check` 时，超时会自动 `check`，并继续推进到下一条街
  - 服务端 `lastAction` 会带上 `auto: true`
  - `check` 的超时动作还会带 `reason: "timeout"`
- 四人连续多手快速回归：
  - 通过连续两手“当前行动玩家直接弃牌”的方式完成多手推进
  - 两次结算后都能自动进入下一手
  - 结果弹窗会在下一手开始时自动清掉
  - 房间总筹码能保持不变
- 重连后退出清理：
  - 用相同 `isolatedContext` 重连后执行 `退出房间`
  - 服务端会正确把对应玩家移出座位并转移房主
  - 剩余最后一位玩家再退出时，房间会正常删除
- 弃牌后补码的真实边界：
  - 玩家在本手中已经弃牌，但牌局尚未结束时，前端仍可打开补码弹窗
  - 实际提交会被服务端拒绝，并返回“当前牌局中无法补码，请先弃牌”
  - 说明真实规则是“本手必须结束”，不是“只要本玩家已弃牌即可”
- 三人真实边池：
  - 短码 `1000`
  - 中码 `2000`
  - 深码 `4000`
  - 动作为：
    - 短码 `all-in 1000`
    - 中码 `all-in 1990`
    - 深码 `call 1980`
- 同一 `isolatedContext` 的关闭后重开：
  - `deviceId` 保留
  - 房间身份恢复
  - 可以作为真实浏览器重连链路来测
- 行动中断线：
  - 当前行动玩家关闭页面后，牌局会自动推进
  - 服务端会把该玩家按断线处理并自动弃牌
- 中途加入第 4 名玩家：
  - 牌局进行中加入会提示“您已入座，将在下一轮游戏开始时加入”
  - 当前手牌不会直接入池
  - 下一手开始后会正常收到手牌并参与盲注/行动
- 满座观战：
  - 真实 `2` 人房中，第 3 名玩家加入会进入观战模式
  - 页面会提示“座位已满，您正在观战模式”
  - 顶部状态会显示 `3/2 (2座 1观)`
- 观战 -> 入座：
  - 已有入座玩家点击“离座观战”后，观战玩家可以点击空位入座
  - 服务端 seat 变更确实发生
- 观战退出房间：
  - 点击“退出房间”后，服务端房间玩家列表能正确移除该玩家
- 房主退出后的房主转移：
  - 未开局房间里，原房主退出后，剩余玩家会自动接任 `isHost`
  - 新房主页面可以正常看到皇冠和“开始游戏”
- 已入座玩家换座：
  - 已入座玩家可以直接点击空位换座
  - 页面位置和服务端 `seat` 都会同步更新
- 房主转移后的新房主开局：
  - 新房主可以正常开局
  - 盲注、手牌、行动权都会正常下发到剩余玩家
- 已开局房间的最后玩家退出：
  - 两名玩家都在已开局房间里依次点击“退出房间”
  - 最后一名玩家退出后，房间会被真正删除
- `Straddle` + 双跑配置：
  - `allowStraddle=true` 和 `allinDealCount=2` 的房间可以成功创建
  - 服务端调试接口能正确反映这两个设置

### 0.4 `chrome-devtools MCP` 的已知现象

- 第二位玩家加入后，房主页有时不会立刻把人数刷到 `2/6`
- 手动 `reload` 后状态显示正确
- 这更像项目自身的实时同步/刷新时机问题，不是 MCP 工具链本身的问题
- 某些按钮或 modal 会在页面自动更新后立刻失效
- 如果点击报 “element no longer exists”，先重新 `snapshot`，不要沿用旧的 uid
- `CreateRoomModal` 里的 `SliderInput` 在 MCP 下直接改 DOM value 不等于 React 状态真的更新
- 如果要精确建 `2` 人房，不要依赖单纯的 DOM 改值；优先：
  - 驱动真实的组件事件
  - 或直接用本地 socket 创建测试房间

### 0.5 `chrome-devtools MCP` 的服务启动建议

不要默认假设“当前对话里临时用 shell 拉起的本地服务”一定能被 MCP 浏览器通过同一个 `localhost` 访问到。

经验结论：

- 最稳的是在真实环境里先把 `server` 和 `client` 启好，再开始 MCP 测试
- 如果 MCP 浏览器里访问 `127.0.0.1` 报 `ERR_CONNECTION_REFUSED`，而 shell 日志又显示服务已经启动，不要立刻重复试很多次
- 先确认服务是不是独立于当前沙箱 shell 存活，必要时由用户手动启动，或用外部稳定脚本启动

### 0.6 `chrome-devtools MCP` 的页面操作建议

- 每次状态变化后立刻重新 `snapshot`
- 不要长期持有旧 uid
- 多玩家场景优先用 `pageId + isolatedContext` 管理，而不是靠页面顺序猜
- 先把页面上下文切到目标玩家，再操作，避免误点错页
- 如果某个 UI 控件明显是 React 受控组件，不要把“DOM value 变了”误判成“业务状态真的变了”
- 最终要以：
  - 页面跳转结果
  - 房间状态
  - `/api/debug/rooms/:roomId`
  这三类信号交叉验证

### 0.7 `chrome-devtools MCP` 下的重连测试建议

已验证现象：

- 关闭某个玩家页面
- 再用同一个 `isolatedContext` 打开相同房间 URL
- `localStorage` 中的 `texas-holdem-device-id` 会保留
- 页面会回到原玩家身份和原房间

结论：

- 这条链路适合做“真实浏览器重连恢复”测试
- 不需要再额外造 profile 目录
- 如果断开时玩家正处于行动中，重连回来时不要期待继续保留行动权
- 当前项目会优先保证牌局推进，断线玩家会被自动弃牌

### 0.8 需要配合调试接口一起看

已验证有些问题只看前端页面容易误判。

建议：

- 关键场景结束后调用：
  - `/api/debug/rooms/:roomId`
- 用它确认：
  - `seat`
  - `chips`
  - `isSpectator`
  - `isActive`
  - `socketId`
  - `gameStarted`

这样能快速判断是：

- 服务端状态错了
- 还是前端没刷出来

### 1. 不优先使用 `agent-browser` 的多 session daemon

实践结果：

- `agent-browser --session host/...` 在当前环境里不稳定
- 很容易出现 `Daemon failed to start`
- 不同 session 名称会留下不同的 `pid/port` 状态文件，排障成本高

稳定替代：

- 手动启动真实 Chrome
- 每个浏览器固定一个 `--remote-debugging-port`
- 用 `agent-browser --cdp <固定端口>` 控制它

- 权限申请命令前缀稳定
- 排障时不用反复确认当前连的是哪一个浏览器

## 已踩过的坑

### 1. `vite dev` / `vite build` 在沙箱里可能报 `spawn EPERM`

表现：

- `npm run dev`
- `npm run build`

都可能在 `esbuild` 子进程阶段失败。

结论：

- 这不是项目前端代码本身的直接错误
- 是当前环境对子进程/沙箱的限制

建议：

- 联机回归时优先使用后端直接托管的 `client/dist`
- 如果必须重新构建，尽量在沙箱外执行

### 2. 后端如果在沙箱内后台启动，真实 Chrome 可能访问不到

表现：

- 日志里看起来像启动了
- 但浏览器访问 `127.0.0.1:3001` 仍然 `ERR_CONNECTION_REFUSED`

原因：

- 子进程没有以真实可访问的方式留在外部环境里

建议：

- 真实浏览器回归时，把后端放到沙箱外启动

### 2.1 不要在同一桌的脏状态上连续拼接多个不同场景

表现：

- 上一手牌的日志、筹码、等待状态会混入下一组验证
- 很容易把“旧状态污染”误判成“新 bug”

建议：

- 高价值场景单独开新房间
- 或至少关闭旧页面，重新用新 `isolatedContext` 进房

### 2.2 测 `maxPlayers` 时，优先用服务端直建测试房

表现：

- `SliderInput` 在 MCP 下容易出现：
  - DOM 的 range 值变了
  - 但 React 的 `settings.maxPlayers` 没变
  - 最终房间还是按默认 `6` 人创建

建议：

- 如果测试重点是“满座观战/房间人数上限”，直接用本地 `socket.io-client` 创建测试房
- 这样能把重点放在真正的房间行为，而不是反复和滑块自动化较劲

### 3. PowerShell 会吞掉 `@e1`

表现：

- `agent-browser click @e1` 在 PowerShell 下可能被错误解析

建议：

- 优先用语义定位：
  - `find role button click --name "..."`
  - `find label "..."`
- 或直接用 `screenshot --annotate`

### 4. `snapshot -i` 抓不到自定义卡片座位区域

表现：

- 页面明明有大量可点区域
- `snapshot -i` 只返回极少数按钮

建议：

- 改用 `snapshot -i -C`
- 或 `screenshot --annotate`

### 5. `get text body` 不稳定时，用 `eval 'document.body.innerText'`

表现：

- `get text body` 有时返回空
- 但页面实际已经正常渲染

建议：

```powershell
agent-browser --cdp 9230 eval 'document.body.innerText'
```

### 6. 结果弹窗会干扰下一手操作

已在代码中修复：

- 新一手开始后自动关闭旧结果弹窗

但经验上仍建议：

- 当页面操作异常时，先确认是否有残留 modal 覆盖层

### 6.1 结果弹窗的文案对边池不够清晰

已验证现象：

- 三人真实边池场景里，结果弹窗会把多个赢家并列显示在同一行
- 这在“主池归一人、边池由另外两人平分”的情况下，语义并不清楚

结论：

- 结算数字本身是对的
- 但结果弹窗需要更明确地区分：
  - 主池赢家
  - 边池赢家
  - 平分金额

建议优化：

- 结果弹窗按 pot 维度展示
- 每个 pot 单独列出参与者、赢家和分配金额

### 6.2 边池的实时展示可读性不够

已验证现象：

- all-in 过程中页面顶部会显示多个 “边池”
- 但用户很难一眼看懂当前哪些数字对应主池，哪些对应边池

建议优化：

- 明确展示：
  - 主池金额
  - 边池 1 金额和参与者
  - 边池 2 金额和参与者
- 不要只显示数字，最好带参与玩家范围

### 6.3 观战玩家入座后，服务端没有补回初始筹码

已验证现象：

- 满座观战玩家在空位出现后点击入座
- 服务端 `seat` 会正确切换到空位
- 但 `/api/debug/rooms/:roomId` 显示该玩家 `chips = 0`

结论：

- 这不是单纯的 UI 刷新问题
- 是真实的座位流转 / 观战转入座筹码初始化 bug

建议优化：

- 观战玩家第一次从 `seat = -1` 入座时，应明确初始化为房间默认筹码
- 或者沿用该玩家已有筹码，但不能保持 `0`

### 6.4 非房主可以开始游戏

已验证现象：

- 服务端 `RoomManager.startGame(roomId, playerId)` 只检查：
  - 玩家存在
  - 玩家不是观战
  - 玩家已入座
  - 入座人数 >= 2
- 没有检查 `player.isHost`

结论：

- 这是真实权限 bug
- 非房主入座玩家可以直接触发开局

建议优化：

- 后端 `startGame` 必须校验 `player.isHost`
- 前端“开始游戏”按钮也应只对房主显示

### 6.5 非法开局会触发前端崩溃

已验证现象：

- 非房主从一个异常房间状态触发 `startGame`
- 页面 console 报错：
  - `TypeError: Cannot read properties of null (reading 'currentBet')`
- 崩溃点在 `ActionButtons`

结论：

- 前端对非法或未完成初始化的 `gameState` 缺少防御

建议优化：

- `ActionButtons` 对 `gameState`、`currentPlayer`、`currentBet` 做空值保护
- 即使后端状态异常，也不应把整页打崩

### 6.6 房主转移后，非房主页面仍显示“开始游戏”

已验证现象：

- 原房主退出后，服务端 `isHost` 已正确转移给下一位玩家
- 但其他非房主入座玩家页面上，仍然能看到“开始游戏”按钮

结论：

- 这是前端权限显示 bug
- 即使服务端 host 已转移，前端按钮显隐仍未严格绑定 `isHost`

建议优化：

- “开始游戏”按钮只对当前 `isHost=true` 的玩家显示
- 房主转移后，按钮显隐应跟随最新房间状态立即刷新

### 6.7 正常双人盲注也会生成 `sidePots`

已验证现象：

- 两人房正常开局、仅有小盲和大盲时

结论：

- 这是规则层 bug
- 普通盲注差额不应该被建成边池

建议优化：

- 只有在真实不等额 all-in 导致资格玩家集合收缩时才创建边池
- 开局后的普通盲注状态下，`sidePots` 应为空数组

### 6.8 离线未退房的玩家会留下脏房间

已验证现象：

### 6.10 `Straddle` 被系统自动打出，不是玩家可选动作

### 6.15 断线自动弃牌后，房间可能卡在 `waiting`

- 当前“断线自动弃牌 -> 结算 -> 重连 -> 下一手启动”这条链路没有完全收口

建议优化：

- 在结算完成和玩家重连后，补一次“是否满足下一手启动条件”的检查
- 对 `waiting` 状态增加兜底恢复逻辑，避免无限停住

### 6.16 “弃牌后可补码”这条文案与真实规则不一致

建议优化：

- 如果服务端策略是不允许“本手中补码”，前端应明确写成：
  - “只能在当前手牌结束后补码”
- 并且在本手未结束时不要展示可提交的补码弹窗

### 6.17 补码弹窗会跨到下一手继续保留

  - `auto: true`
  - `reason: "leave_seat"`
- 房主离座后，房间前端显示为 `7/6 (5座 2观)`，观战者页面也能看到空位
- 观战者点击空位后，前端提示：

- 但服务端真实状态变成：
  - 原房主：`seat = -1`
  - 新观战者：`seat = 0`
  - 新观战者：`chips = 0`

- 新入座玩家在真正生效前，不应出现在：
  - `dealerIndex`
  - `smallBlindIndex`
  - `bigBlindIndex`
  - `currentPlayerIndex`
- 新入座玩家生效时，必须带上正确初始筹码，不能以 `0` 筹码落到正式座位上
- 观战转入座后的 UI 和调试接口都应以同一状态源为准，避免页面显示“已入座”，服务端却还是半观战半入座状态

- 这不是“按钮被 UI 隐藏了，其实手动调 API 就能恢复”
- 当前实现里，脏房状态下的 `startGame()` 实际上只是静默无效

建议优化：

- 对 `gameStarted = true && phase = waiting && currentPlayerId = null` 这类状态，服务端不能再把 `startGame()` 当普通无事发生处理
- 至少应：
  - 明确返回错误
  - 或主动执行修复逻辑

### 6.45 即使房间非常脏，所有在线玩家逐个 `leaveRoom` 后，服务端最终仍能把房间完整删除

房间：

- `38MRGR`

已验证现象：

- 当时房间里同时存在：
  - 2 名健康在座玩家
  - 1 名补码后重新入座的玩家
  - 2 名观战/脏数据玩家

  - `leaveRoom('38MRGR')`
- 最终调试接口返回：
  - `404`
  - `房间不存在`

结论：

- 虽然 `38MRGR` 这种脏房不会自愈
- 但“所有在线玩家依次离开 -> 服务端删房”的最终清理链路仍然是通的

建议优化：

- 如果短期内无法完整修复脏房自愈，至少应保留这条可靠的清理出口
- 前端也最好能在房间明显不可恢复时给出更明确的“退出并重建房间”提示
