# 真实浏览器联机回归踩坑记录

这份文档记录本项目在 Codex + `agent-browser` / `chrome-devtools MCP` + 真实 Chrome 联机回归时踩过的坑、稳定做法和后续执行规范。

## 目标

- 避免重复走弯路
- 避免因为 `--cdp` 端口递增而反复申请相似权限
- 固化一套可重复执行的真实浏览器回归流程

## 新增固定坑

### `manage-real-browser-env.ps1 status` 不等于你手动开的 `pnpm dev` 已失效

已验证现象：

- 用户本地手动开了：
  - `server`: `pnpm dev` -> `3001`
  - `client`: `pnpm dev` -> `5173`
- 这时运行：
  - `pwsh -NoProfile -File scripts/manage-real-browser-env.ps1 status`
- 输出只会告诉你 dedicated regression pair 里的：
  - `3101`
  - `5173`
- 所以很容易出现一种误判：
  - `status` 看起来“不健康”
  - 但其实当前 checkout 的本地 dev 环境已经可以直接用于浏览器验证

稳定做法：

- 如果你明确知道自己是手动开的本地 dev，不要因为 `status` 里没看到 `3101` 就立刻重启
- 先查：
  - `GET http://127.0.0.1:3001/api/debug/devices`
  - `GET http://127.0.0.1:5173/`
- 只有当：
  - `3001` 不通
  - 或 `5173` 指向了错误后端 / 错误 checkout
  - 或环境明显脏掉
  才切回 runbook 的 `start-all`

结论：

- `status` 只对 runbook 的 `3101 / 5173` 回归对口负责
- 手动本地 dev 的健康度，需要单独核对 `3001`

### 手机 support sheet 看起来对，不代表页面真的不再抢滚动

已验证现象：

- 初版 single-screen terminal 即使视觉上已经像 bottom sheet：
  - `Roster / Hand Tape / Room` 都能正常弹出
- 但真实浏览器里打开 sheet 后再测：
  - `document.scrollingElement.scrollHeight > clientHeight`
- 结果就是：
  - 页面底层还留着一小段隐藏纵向滚动
  - 手感上会出现“sheet 在滚、页面偶尔也跟着抢一下”的 scroll fight

稳定做法：

- 手机 sheet 回归不能只看截图
- 每次至少补一条 live 检查：
  - `document.scrollingElement.scrollHeight`
  - `document.scrollingElement.clientHeight`
  - `.room-panel-sheet__body.scrollHeight / clientHeight`
- 目标应是：
  - 页面本身 `pageScrollable = false`
  - sheet 内容区 `sheetScrollable = true`

这轮实测结果：

- 房间 `9219U5`
- `390x844`
- 打开 `Roster` 后：
  - `pageScrollable = false`
  - `sheetScrollable = true`
  - `shellOverflow = hidden`
  - `contentOverflow = hidden`

结论：

- 手机 single-screen terminal 的验收标准不是“看起来像单屏”
- 而是：主页面真不滚，只让 sheet 自己滚

### 手机 seat ring 不能按“猜测高度”做几何预算

已验证现象：

- 第一版 mobile `seatRingLayout` helper 把手机 seat 卡片预算成了大约 `70 x 60`
- 但真实浏览器里，`arena-seat-card` 实际占位大约是：
  - `70 x 123`
  - `70 x 128`
- 结果就是：
  - 单元测试先过了
  - `390x844` 真机视图里，上下 seat 仍然压进桌面

稳定做法：

- 手机 seat 几何不要靠“看起来差不多”的假设
- 至少量一次真实浏览器里的：
  - `.arena-seat-anchor`
  - `.poker-table`
- 再把 helper 的 `cardHeight / gap` 调整到和真实占位一致

结论：

- 这类响应式几何问题，Node 单测只能兜底，不能代替真实 DOM 尺寸核对
- 如果 helper 预算和实际 DOM 不一致，测试照样可能给出假绿灯

### 手机桌面不仅要看 seat ring，还要看公共牌带

已验证现象：

- 修完 `seat ring vs table circle` 后，手机页脚本已经是：
  - `overlappingTableSeats = 0`
- 但真实截图仍然能看到左右座位压到公共牌区域
- 根因不是 seat ring 又坏了，而是：
  - `5` 张 `community` 牌在手机下仍沿用桌面尺寸
  - 公共牌带宽度超出了窄桌面的安全宽度

稳定做法：

- 手机 responsive 几何至少检查两层：
  - `seat vs table`
  - `seat vs community-card band`
- 把公共牌尺寸、间距、tray 宽度也收进 layout helper
- 手机下必要时：
  - 缩小 community card footprint
  - 隐藏非关键 phase 标题
  - 取消不必要的额外 padding

结论：

- “桌面圆盘没重叠”不等于手机桌面真的可用
- 对窄屏牌桌，公共牌带本身就是第二条硬约束

### Tailwind 4 官方升级器在当前仓库路径下可能直接失败

已验证现象：

- 在当前仓库路径 `D:\\GITHUB\\Texas-hold'em` 下执行：
  - `cd client`
  - `npx @tailwindcss/upgrade -f`
- 升级器会在加载 `tailwind.config.js` 时失败，报：
  - `Could not load the configuration file: Unterminated string: ';'`
- 同一个 `tailwind.config.js` 用 Node 直接 `import()` 是可以正常加载的

稳定做法：

- 不要把这次失败误判成配置文件本身损坏
- 当前仓库路径里带单引号，优先怀疑升级器对路径处理不稳
- 在这个仓库里升级 Tailwind 4 时，优先采用手工迁移并结合官方升级指南核对

结论：

- 当前仓库下，`@tailwindcss/upgrade` 不是稳定路径
- 先手工迁移、再跑构建和浏览器回归，更可控

### Tailwind 4 会直接拦下 v3 已废弃 utility，不能等页面肉眼发现

已验证现象：

- 迁到 `tailwindcss@4.2.2` 后，第一次 `npm run build` 会直接失败：
  - `Cannot apply unknown utility class 'bg-opacity-50'`
- 同一轮还发现一批需要同步替换的旧写法：
  - `flex-shrink-0 -> shrink-0`
  - `backdrop-blur-sm -> backdrop-blur-xs`

稳定做法：

- Tailwind 4 迁移后，先全量 grep 旧 utility 再构建
- 至少检查：
  - `bg-opacity-*`
  - `text-opacity-*`
  - `border-opacity-*`
  - `flex-shrink-*`
  - `flex-grow-*`
  - `backdrop-blur-sm`
  - 其他官方 upgrade guide 中点名的 renamed / removed utility

结论：

- 不要把 Tailwind 4 迁移理解成“升级依赖后等视觉回归再说”
- 第一轮就应该主动清旧 utility，否则构建和视觉都不可靠

### 合并回 `main` 后，不要继续照抄旧 worktree 路径

已验证现象：

- 产品化分支已经合并回 `main` 后，历史 runbook/README 里仍可能残留 `.worktrees/presentation-state-refactor`
- 这会让第一条环境命令就直接指向不存在的路径

稳定做法：

- 每次进入新阶段前，先确认当前真实执行根目录
- 当前标准根目录应为：
  - `D:\\GITHUB\\Texas-hold'em`
- 当前真实浏览器回归脚本路径应为：
  - `D:\\GITHUB\\Texas-hold'em\\scripts\\manage-real-browser-env.ps1`
  - `D:\\GITHUB\\Texas-hold'em\\scripts\\browser-room-workflow.ps1`

结论：

- 分支合并或 worktree 移除后，先修文档路径，再跑环境命令
- 如果文档里的根目录和当前仓库状态不一致，不要硬跑，先改文档

### 创建房间不能继续用 fire-and-forget

已验证现象：

- 旧实现里 `CreateRoomModal` 提交后会立即关闭
- 但 `createRoom` 当时并不等待服务端确认是否真的建房成功
- 这会导致失败时：
  - 用户先丢失上下文
  - `isCreatingRoom` 可能停在错误状态
  - 再看见错误 toast 时，已经回不到原提交界面

稳定做法：

- 创建房间必须走明确的成功 / 失败 / 超时链路
- 模态框只在收到 `roomCreated` 后关闭
- 失败时保持模态框开启，允许用户直接修正后重试

结论：

- 房间创建属于首个关键动作，不能接受 fire-and-forget
- 这类关键路径后续都应逐步改成可等待、可超时、可精确报错的请求语义

### 关键 socket 操作不能共用一个松散的 `error` 通道

已验证现象：

- `joinRoom` 和 `requestRebuy` 这类关键操作原先依赖共享的 socket `error` 事件
- 这意味着只要时机撞上，理论上就可能出现“不是这个请求的错误，却把这个请求 Promise 拒掉”的串味风险

稳定做法：

- 关键请求优先使用作用域明确的成功 / 错误事件
- 例如：
  - `joinRoom` 对 `joinedRoom / joinRoomError`
  - `requestRebuy` 对 `rebuySuccess / requestRebuyError`

结论：

- 共享 `error` 更适合作为全局 toast 或兜底信号
- 关键用户动作应逐步改成自己的 request/response 语义，不要继续混在线路里

### 房间模式和显示模式不是一回事

已验证现象：

- `roomMode` 是共享房间设置，所有玩家都应看到同一个房间语义
- `displayModePreference` 是本地偏好，只影响当前客户端的信息密度和展示方式
- 如果把两者混在一起，后面很容易做出“本地切个模式，结果改了整桌规则”的错误实现

稳定做法：

- 房间创建时只提交 `roomMode`
- 本地持久化只保存 `displayModePreference`
- 实际显示使用：
  - `effectiveDisplayMode = resolveDisplayMode(roomMode, displayModePreference)`

结论：

- `roomMode` 负责共享语义
- `displayModePreference` 负责本地展示
- 之后每次加模式相关功能，都先判断它属于哪一层

### 入座/换座/离座的提示必须在服务端确认后再发

已验证现象：

- 旧实现里：
  - `takeSeat` 会先发提示，再等服务端
  - `leaveSeat` 会在客户端先额外 `fold` 再发离座
- 这会产生两类问题：
  - 失败时用户已经看到了错误的成功提示
  - 离座链路里会出现不必要的重复动作和时序竞争

稳定做法：

- seat 相关关键动作统一改成：
  - 服务端返回明确结果
  - 前端 `await` 成功结果
  - 再根据返回的 `tableState / forcedFold / roomState` 生成提示

结论：

- 对 seat 类关键动作，广播状态不是用户反馈语义
- 真正的用户提示应建立在服务端确认返回值上

### `ROOM_RECOVERY_REQUIRED` 不能既当专用状态又当普通失败各报一遍

已验证现象：

- `startGame` 在脏房场景下会触发一个真正有业务语义的分支：
  - 房间状态切到 `recovery_required`
  - 客户端收到专门的 `roomRecoveryRequired` 事件和恢复 banner
- 如果这时前端还把同一次失败当作普通按钮错误再弹一遍：
  - 用户会同时看到“等待恢复”的专用提示
  - 又看到“开始游戏失败”的通用错误 toast
- 两条提示语义重复，而且一个像引导，一个像异常，容易让人误判是否真的出错

稳定做法：

- 关键 request/response 错误需要保留结构化 `code`
- `emitWithResponse` 不能只保留 `message`
- `startGame` 这种动作在 `code = ROOM_RECOVERY_REQUIRED` 时，应让专用恢复事件负责 UI 提示，避免额外重复 toast

结论：

- “需要恢复”不是普通报错，而是一个专门的产品状态
- 后续凡是存在专用 banner / 状态迁移的异常，都不要再用第二条通用 toast 重复表达同一件事

### 点击“开始游戏”不等于一定真的开始了一手牌

已验证现象：

- 服务端接受 `startGame` 后，仍然可能在 `startNewHand()` 阶段发现当前没有足够可参战玩家
- 这时房间会回到干净的 `idle`
- 如果前端把“按钮点击被接受”直接当成“本手已经开始”，提示就会误导用户

稳定做法：

- `startGame` 应返回明确的 `handStarted / roomState`
- 前端反馈必须基于这个返回值，而不是基于按钮点击本身
- 当 `handStarted = false` 且 `roomState = idle` 时，应提示“房间已回到等待开始状态”，而不是假装已经开局

结论：

- 对关键生命周期动作，`success accepted` 和 `state actually advanced` 不是一回事
- 房间级动作需要服务端返回最终语义，前端不能靠乐观假设补全

### 行动按钮不能把“5 秒后自动解锁”当成异常处理

已验证现象：

- 旧实现里 `ActionButtons` 点完 `fold / call / raise / allin` 后会先锁住 UI
- 一旦服务端拒绝这次操作，比如：
  - 已经不是你的回合
  - 玩家当前无法行动
  - 参数不合法
- 前端并没有这次请求自己的专用失败事件
- 结果就是：
  - 用户只能等共享 `error` toast
  - 按钮本身往往要等到 5 秒 fallback timeout 才重新解锁

稳定做法：

- `playerAction` 要有明确的 `playerActionSuccess / playerActionError`
- 操作区的锁定状态应在请求失败时立即释放，而不是依赖超时保护兜底
- `revealHand` 这类 showdown 选择也一样，不能继续混在共享 `error` 通道里

结论：

- “防止重复提交”的锁不应该兼任异常恢复机制
- 关键交互要么成功推进状态，要么立刻精确失败，不能让用户卡在半锁定 UI 里等超时

### 同一设备切到新房前，服务端必须先把它从旧房移除

已验证现象：

- 旧实现里，如果同一个 `deviceId`：
  - 先创建房间 A
  - 回到首页但不显式退出
  - 再创建或加入房间 B
- 这个设备会同时残留在旧房和新房
- 后续像 `playerAction` 这类按 `playerId/deviceId` 反查房间的链路，就可能命中旧房
- 真实浏览器里表现为：
  - 新房页面已经显示正在进行的手牌
  - 但点击行动后却收到 `操作失败：游戏未开始`

稳定做法：

- 服务端在同一设备创建或加入新房前，先把它从其他房间清掉
- 对这类问题，不要只看前端页面，要同时核对：
  - `GET /api/debug/rooms/<oldRoomId>`
  - `GET /api/debug/rooms/<newRoomId>`

结论：

- “同一设备同时残留在多个房间”会直接破坏按玩家反查当前房间的正确性
- 这不是单纯 UI 提示问题，而是房间归属根因问题

### 切房后如果页面短暂显示旧房残影，先看调试接口，不要先猜错层

已验证现象：

- 之前有一轮真实浏览器回归里：
  - 从旧房回首页
  - 立刻创建新房
- 新房页面曾短暂显示旧房玩家列表和旧牌局记录
- 但同一时刻 `GET /api/debug/rooms/:roomId` 已经显示新房服务端状态是干净的
- 刷新页面后，前端也回到了正确的新房状态

稳定做法：

- 遇到“切到新房后 UI 看起来像旧房没清干净”时，先分层判断：
  - 先查服务端 debug 房间接口
  - 再决定这是服务端成员归属问题，还是客户端本地状态残影
- 如果服务端是干净的，不要立刻把问题误记成 gameplay 引擎脏状态

结论：

- 切房残影问题至少有两层：
  - 服务器成员归属是否正确
  - 客户端本地房间状态是否及时重置
- 调试时先分层，不然很容易把真正根因记错

### `handHistory` 里的公共牌和亮牌不是字符串，不能直接 `join(' ')`

已验证现象：

- `pro` 模式新增 `Board ...` 行后，真实浏览器一度显示：
  - `[object Object] [object Object] ...`
- 调试接口里 `handHistory.communityCards` 的真实形态是：
  - `{ rank, suit }`
- `reveals[].cards` 也是同样的数据形态

稳定做法：

- `handHistory` 和 reveal 摘要都必须先把 card object 显式格式化成：
  - `A♠`
  - `K♥`
  - `10♣`
- 不能假设这些卡牌在进入 view-model 前已经变成字符串

结论：

- 这是一个典型的“浏览器 UI 看起来是字符串，实际 payload 是对象”的坑
- 后续凡是给历史记录、结算摘要、study 模式回看加牌面文本，都先确认 card payload 形态

## 已验证的稳定方案

### 0. 如果目标是“更少的同意操作”，优先用 `chrome-devtools MCP`

实践结果：

- 当前会话里，`chrome-devtools MCP` 一旦配置完成并重启生效，后续浏览器操作不再依赖新的 shell 命令
- 在 `server` 和 `client` 已经由真实环境启动好的前提下，打开页面、切页、点击、填表、查看 console/network 基本不需要新的同意
- 这比 `agent-browser` 的 CLI 流程更省审批

结论：

- 想要最低审批成本时，优先用 `chrome-devtools MCP`
- 想要继续复用固定 shell 脚本工作流时，用 `agent-browser`

### 0.1 `chrome-devtools MCP` 安装和配置要点

配置位置：

- [config.toml](/C:/Users/lenovo/.codex/config.toml)

当前有效配置：

```toml
[mcp_servers.chrome-devtools]
command = "cmd"
args = [
    "/c",
    "npx",
    "-y",
    "chrome-devtools-mcp@latest",
]
env = { SystemRoot = "C:\\Windows", PROGRAMFILES = "C:\\Program Files" }
startup_timeout_ms = 20_000
```

注意：

- 改完 `config.toml` 后必须重启 `Codex` 会话，新的 MCP 工具才会真正出现在工具列表里
- 只验证 `npx chrome-devtools-mcp@latest --help` 不够，最终要以“会话里已经出现 `chrome-devtools` 工具”为准

### 0.2 `chrome-devtools MCP` 做多玩家联机时，不能只开普通标签页

已验证现象：

- 同一个浏览器上下文里的多个标签页会共享 `localStorage`
- 本项目用 `deviceId` 识别玩家身份
- 所以如果只开第二个普通标签页，服务端会把它识别成同一个玩家，不会形成真实双人联机

稳定做法：

- 第一个页面用默认上下文创建房间
- 第二个页面必须使用 `isolatedContext`
- 示例：`isolatedContext = player2`

结论：

- `chrome-devtools MCP` 里模拟多玩家，核心不是“多页”，而是“多隔离上下文”

### 0.2.1 不要复用旧的 `isolatedContext` 做新场景

已验证现象：

- 同一个 `isolatedContext` 会保留 `localStorage`
- 本项目的设备身份保存在 `localStorage['texas-holdem-device-id']`
- 如果复用旧上下文继续做新房间或新场景，旧房间状态、旧筹码、旧身份都会污染当前回归

稳定做法：

- 每组回归都使用全新的上下文名
- 示例：
  - `host-mcp-1`
  - `guest-mcp-1`
  - `host-mcp-3`
  - `p2-mcp-3`
  - `p3-mcp-3`

建议：

### 0.5 `chrome-devtools MCP` 的服务启动建议

不要默认假设“当前对话里临时用 shell 拉起的本地服务”一定能被 MCP 浏览器通过同一个 `localhost` 访问到。

经验结论：

- 最稳的是在真实环境里先把 `server` 和 `client` 启好，再开始 MCP 测试
- 如果 MCP 浏览器里访问 `127.0.0.1` 报 `ERR_CONNECTION_REFUSED`，而 shell 日志又显示服务已经启动，不要立刻重复试很多次

### 0.6 `chrome-devtools MCP` 的页面操作建议

- 多玩家场景优先用 `pageId + isolatedContext` 管理，而不是靠页面顺序猜
- 先把页面上下文切到目标玩家，再操作，避免误点错页
- 如果某个 UI 控件明显是 React 受控组件，不要把“DOM value 变了”误判成“业务状态真的变了”
- 最终要以：
  - 页面跳转结果
  - 房间状态
  - `/api/debug/rooms/:roomId`
  这三类信号交叉验证

## 已踩过的坑

### 1. `vite dev` / `vite build` 在沙箱里可能报 `spawn EPERM`

表现：

- `npm run dev`
- `npm run build`

都可能在 `esbuild` 子进程阶段失败。

结论：

- 这不是项目前端代码本身的直接错误
- 是当前环境对子进程/沙箱的限制

建议：

- 联机回归时优先使用后端直接托管的 `client/dist`
- 如果必须重新构建，尽量在沙箱外执行

### 2. 后端如果在沙箱内后台启动，真实 Chrome 可能访问不到

表现：

- 日志里看起来像启动了
- 但浏览器访问 `127.0.0.1:3001` 仍然 `ERR_CONNECTION_REFUSED`

原因：

- 子进程没有以真实可访问的方式留在外部环境里

建议：

- 真实浏览器回归时，把后端放到沙箱外启动

### 2.1 不要在同一桌的脏状态上连续拼接多个不同场景

表现：

- 上一手牌的日志、筹码、等待状态会混入下一组验证
- 很容易把“旧状态污染”误判成“新 bug”

建议：

- 高价值场景单独开新房间
- 或至少关闭旧页面，重新用新 `isolatedContext` 进房

### 2.2 测 `maxPlayers` 时，优先用服务端直建测试房

表现：

- `SliderInput` 在 MCP 下容易出现：
  - DOM 的 range 值变了
  - 但 React 的 `settings.maxPlayers` 没变
  - 最终房间还是按默认 `6` 人创建

建议：

- 如果测试重点是“满座观战/房间人数上限”，直接用本地 `socket.io-client` 创建测试房
- 这样能把重点放在真正的房间行为，而不是反复和滑块自动化较劲

### 3. PowerShell 会吞掉 `@e1`

表现：

- `agent-browser click @e1` 在 PowerShell 下可能被错误解析

建议：

- 优先用语义定位：
  - `find role button click --name "..."`
  - `find label "..."`
- 或直接用 `screenshot --annotate`

### 4. `snapshot -i` 抓不到自定义卡片座位区域

表现：

- 页面明明有大量可点区域
- `snapshot -i` 只返回极少数按钮

建议：

- 改用 `snapshot -i -C`
- 或 `screenshot --annotate`

### 5. `get text body` 不稳定时，用 `eval 'document.body.innerText'`

表现：

- `get text body` 有时返回空
- 但页面实际已经正常渲染

建议：

```powershell
agent-browser --cdp 9230 eval 'document.body.innerText'
```

### 6. 结果弹窗会干扰下一手操作

已在代码中修复：

- 新一手开始后自动关闭旧结果弹窗

但经验上仍建议：

- 当页面操作异常时，先确认是否有残留 modal 覆盖层

### 6.1 结果弹窗的文案对边池不够清晰

已验证现象：

- 三人真实边池场景里，结果弹窗会把多个赢家并列显示在同一行
- 这在“主池归一人、边池由另外两人平分”的情况下，语义并不清楚

结论：

- 结算数字本身是对的
- 但结果弹窗需要更明确地区分：
  - 主池赢家
  - 边池赢家
  - 平分金额

建议优化：

- 结果弹窗按 pot 维度展示
- 每个 pot 单独列出参与者、赢家和分配金额

### 6.2 边池的实时展示可读性不够

已验证现象：

- all-in 过程中页面顶部会显示多个 “边池”
- 但用户很难一眼看懂当前哪些数字对应主池，哪些对应边池

建议优化：

- 明确展示：
  - 主池金额
  - 边池 1 金额和参与者
  - 边池 2 金额和参与者
- 不要只显示数字，最好带参与玩家范围

### 6.3 观战玩家入座后，服务端没有补回初始筹码

已验证现象：

- 满座观战玩家在空位出现后点击入座
- 服务端 `seat` 会正确切换到空位
- 但 `/api/debug/rooms/:roomId` 显示该玩家 `chips = 0`

结论：

- 这不是单纯的 UI 刷新问题
- 是真实的座位流转 / 观战转入座筹码初始化 bug

建议优化：

- 观战玩家第一次从 `seat = -1` 入座时，应明确初始化为房间默认筹码
- 或者沿用该玩家已有筹码，但不能保持 `0`

### 6.4 非房主可以开始游戏

已验证现象：

- 服务端 `RoomManager.startGame(roomId, playerId)` 只检查：
  - 玩家存在
  - 玩家不是观战
  - 玩家已入座
  - 入座人数 >= 2
- 没有检查 `player.isHost`

结论：

- 这是真实权限 bug
- 非房主入座玩家可以直接触发开局

建议优化：

- 后端 `startGame` 必须校验 `player.isHost`
- 前端“开始游戏”按钮也应只对房主显示

### 6.5 非法开局会触发前端崩溃

已验证现象：

- 非房主从一个异常房间状态触发 `startGame`
- 页面 console 报错：
  - `TypeError: Cannot read properties of null (reading 'currentBet')`
- 崩溃点在 `ActionButtons`

结论：

- 前端对非法或未完成初始化的 `gameState` 缺少防御

建议优化：

- `ActionButtons` 对 `gameState`、`currentPlayer`、`currentBet` 做空值保护
- 即使后端状态异常，也不应把整页打崩

### 6.6 房主转移后，非房主页面仍显示“开始游戏”

已验证现象：

- 原房主退出后，服务端 `isHost` 已正确转移给下一位玩家
- 但其他非房主入座玩家页面上，仍然能看到“开始游戏”按钮

结论：

- 这是前端权限显示 bug
- 即使服务端 host 已转移，前端按钮显隐仍未严格绑定 `isHost`

建议优化：

- “开始游戏”按钮只对当前 `isHost=true` 的玩家显示
- 房主转移后，按钮显隐应跟随最新房间状态立即刷新

### 6.7 正常双人盲注也会生成 `sidePots`

已验证现象：

- 两人房正常开局、仅有小盲和大盲时
- `/api/debug/rooms/:roomId` 已经出现：
  - `sidePots: [{ amount: 10, ... }]`
- 此时并没有任何 `all-in`

结论：

- 这是规则层 bug
- 普通盲注差额不应该被建成边池

建议优化：

- 只有在真实不等额 all-in 导致资格玩家集合收缩时才创建边池
- 开局后的普通盲注状态下，`sidePots` 应为空数组

### 6.8 离线未退房的玩家会留下脏房间

已验证现象：

- 某些异常场景下，房间里只剩离线玩家，且这些玩家没有执行 `leaveRoom`
- `/api/debug/rooms/:roomId` 仍然能查到该房间
- 房间会停留在 `gameStarted=true` 但无人在线的状态

结论：

- 当前清房逻辑是按 `hasLeftRoom` 判断，不是按“是否仍有在线/可恢复玩家”
- 这会留下脏房间

建议优化：

- 增加“全员离线超时清理”策略
- 或在异常停局状态下，允许房主重连后主动重置/关闭房间

### 6.9 正常 `leaveRoom` 的最终删房是通的

已验证现象：

- 已开局房间里，玩家依次点击“退出房间”
- 最后一名玩家退出后，再访问 `/api/debug/rooms/:roomId` 返回 `404`

结论：

- 正常退房路径上的删房逻辑是成立的
- 当前主要问题不是“不会删房”，而是“离线未 leave 的残留房间”

### 6.10 `Straddle` 被系统自动打出，不是玩家可选动作

已验证现象：

- 房间设置 `allowStraddle=true`
- 三人桌开局后，系统会直接产生一条：
  - `lastAction.label = "straddle"`
  - `lastAction.auto = true`
- 没有看到玩家侧的 `Straddle` 选择动作

结论：

- 当前实现不是“允许 straddle”，而是“自动强制 straddle”
- 这与常规德州语义不一致

建议优化：

- `allowStraddle` 只应开放一个可选动作
- 应由符合位置条件的玩家主动选择是否 straddle
- 不应在没有玩家输入时自动代打

  - `reason: "leave_seat"`

  - `smallBlindIndex`
  - `bigBlindIndex`
  - `currentPlayerIndex`
- 新入座玩家生效时，必须带上正确初始筹码，不能以 `0` 筹码落到正式座位上
- 观战转入座后的 UI 和调试接口都应以同一状态源为准，避免页面显示“已入座”，服务端却还是半观战半入座状态

## 当前可优化项

### 前端交互和展示

- 房间成员变化后的实时刷新仍不够稳定，部分页面需要手动 `reload`
- 结果弹窗对边池赢家展示不清晰
- 顶部主池/边池展示可读性不足
- 长时间联机后，日志区域会积累很多历史动作，后续可以考虑分页或按手数折叠
- 第 4 名玩家中途加入时，等待下一轮的提示是对的，但 UI 可以更明确显示：
  - 当前是否已入池
  - 下一手何时生效
  - 当前手牌为什么没有两张底牌
- 观战玩家入座后，UI 和服务端都应显示正确筹码，当前为 `0` 是错误行为
- `开始游戏` 按钮应该只对房主显示
- 非法或不完整的 `gameState` 不应导致整个页面白屏
- 普通盲注场景下不应生成 `sidePots`
- `Straddle` 不应自动强制执行
- `Straddle` 日志不应显示“加注至 0”
- `allinDealCount > 1` 时应真正执行多次 run
- 重连后页面不应映射到错误玩家视角
- 断线自动弃牌后的房间不应长期卡在 `waiting`
- 补码规则文案必须与服务端真实限制一致
- 补码弹窗不应跨手牌保留
- 6 人房连续多手虽然能跑，但错误 `sidePots` 仍然每手都会出现
- 日志需要按 `handNumber` 隔离，不能无限混入旧手动作
- 满座观战者中途入座时，不能以 `0` 筹码进入正式座位，更不能被卷入盲注位
- 出现 `seat >= 0 && chips = 0` 的死座位时，服务端应自动修复，不能靠用户手工 `离座观战`
- 前端“我的位置”与服务端真实 `seat` 仍会串位
- 对于全员离线但未 `leaveRoom` 的房间，需要有超时清理策略
- `开始游戏` 的权限必须由服务端强校验，不能只靠前端隐藏按钮
- 房主处于观战态时，是否允许直接开局需要统一规则和文案
- 修 `startGame` 权限时，要确认观战房主仍然不会被错误卷进手牌
- 页面全部丢失后复用同名 `isolatedContext` 不能默认等于“恢复旧身份”
- 需要一个明确的“按已知 `deviceId` 恢复玩家身份”工作流
