# 德州扑克牌局展示与状态重构设计

日期：2026-03-07

状态：已评审通过

## 1. 目标

本次设计的目标不是单纯美化 UI，而是把当前项目从“原型式状态展示”重构为：

- 对局中信息优先、操作不中断
- 结算阶段有克制但高质感的戏剧反馈
- 服务端状态权威，前端不再猜测牌局语义
- 手牌记录可回溯、可解释、可作为争议裁判

产品方向确定为：

- 职业实战型为骨架
- 社交展示为增强
- 结算有质感，但绝不打断牌局节奏

## 2. 产品原则

- 对局阶段：桌面优先，信息优先，不使用打断型结算弹窗
- 结算阶段：使用短暂的结算窗口替代弹窗
- 记录系统：从“游戏日志”升级为“按手牌组织的 Hand History”
- 展示规则：所有亮牌、结算、盈亏、边池解释都以服务端权威状态为准
- 错误处理：非法或脏状态必须显式暴露，不允许 silent no-op

## 3. 一手牌的标准节奏

### 3.1 对局阶段

桌面只显示当前最重要的信息：

- 当前行动玩家
- 当前下注与跟注成本
- 主池与边池
- 公共牌
- 每位玩家的当前筹码、状态标签、当前下注

### 3.2 结算阶段

取消现有 `HandResultModal` 弹窗，改为 `settling` 结算窗口：

- 最后一个动作完成后，服务端确定赢家、主池/边池归属、每位玩家本手净变化
- 进入 3 到 5 秒的结算窗口
- 桌面执行筹码归集动画
- 获胜者座位高亮
- 允许有资格的玩家在窗口期内选择：
  - 不亮牌
  - 亮一张
  - 亮两张
- 倒计时结束后自动开始下一手

设计约束：

- 亮牌行为不能阻塞下一手
- 已亮出的牌不能撤回隐藏
- `show_one` 必须明确亮左牌还是右牌

## 4. 服务端权威状态模型

本次重构建议将当前松散的房间/玩家/UI 推导拆成四层模型。

### 4.1 房间层 `RoomState`

建议状态：

- `idle`：房间已创建，未开局
- `in_hand`：正在进行一手牌
- `settling`：本手结束，正在结算和亮牌窗口
- `recovery_required`：房间状态异常，需要修复或重置
- `closed`：房间已关闭

关键原则：

- 不能继续用单一 `waiting` 同时表达“等开局”“等下一手”“卡死”

### 4.2 手牌层 `HandState`

建议至少包含：

- `handNumber`
- `street`
- `button`
- `smallBlind`
- `bigBlind`
- `currentActor`
- `board`
- `mainPot`
- `sidePots`
- `actionOrder`
- `settlementWindowEndsAt`
- `eligibleRevealPlayers`
- `revealPolicy`

### 4.3 玩家桌内层 `SeatState / PlayerTableState`

建议使用有限状态，而不是继续依赖多个布尔组合判断：

- `spectating`
- `seated_ready`
- `seated_wait_next_hand`
- `active_in_hand`
- `folded_this_hand`
- `all_in_this_hand`
- `disconnected`
- `busted_wait_rebuy`

这样可以直接避免或收敛以下问题：

- 观战者 mid-hand 抢空座
- `0` 筹码死座位
- `waiting` 与 `in_hand` 文案混淆
- 补码入口依赖前端猜测状态

### 4.4 资金台账层 `PlayerLedger`

建议将玩家资金展示与本手结算分开建模：

- `initialBuyIn`
- `rebuyTotal`
- `totalBuyIn`
- `currentChips`
- `sessionNet`
- `handStartChips`
- `handDelta`
- `showdownDelta`

## 5. Hand Record 设计

`drawer` 不再直接消费 `lastAction`，而是消费服务端生成的不可变 `HandRecord`。

建议字段：

- `handNumber`
- `participants`
- `streets`
- `actions`
- `board`
- `pots`
- `showdown`
- `reveals`
- `chipDeltas`
- `winners`

语义要求：

- `actions` 记录这一手发生了什么
- `chipDeltas` 记录这一手每位玩家最终赢/输多少
- 两者不可混用

## 6. Drawer 设计

`drawer` 目标是 Hand History，不是简单日志面板。

### 6.1 结构

- 默认折叠，可展开
- 按 `handNumber` 倒序
- 每手固定分为：
  - `Summary`
  - `Preflop`
  - `Flop`
  - `Turn`
  - `River / Showdown`

### 6.2 每手摘要

摘要应直接解释清楚：

- 谁赢了主池
- 谁赢了边池
- 谁平分了哪个池
- 每位玩家本手净变化
- 谁亮了一张/两张/未亮

示例语义：

- `玩家A 赢得主池 +3000`
- `玩家B 与 玩家C 平分边池，各 +1000`
- `玩家D 本手净输 -1000`
- `玩家B 亮牌：A♠`
- `玩家C 亮牌：K♦ Q♦`
- `玩家A 盖牌`

### 6.3 动作记录

每条动作至少包含：

- 玩家名
- 动作
- 金额
- 总投入或总下注
- 是否自动动作
- 是否 all-in
- 是否亮牌

## 7. Show Hand 规则

本项目采用混合规则。

### 7.1 默认规则

- 默认房间规则为 `showdown_only`
- 只有进入摊牌的玩家才有亮牌权
- 已弃牌玩家在记录中只表现为 `muck/folded`

### 7.2 私人房增强规则

- 房间可开启 `free_reveal_after_hand`
- 本手参与玩家都可以在结算窗口内选择：
  - 不亮
  - 亮一张
  - 亮两张

### 7.3 约束

- 亮牌权限由服务端校验
- 亮牌结果在结算窗口结束时锁定
- 已亮不能撤回
- 观战者只能看，不能操作
- 断线重连到 `settling` 时，需要恢复剩余倒计时和自身 reveal 权限

## 8. 玩家席位与资金展示

### 8.1 席位常驻信息

建议每个席位常驻展示：

- 当前筹码
- 相对总带入净盈亏

例如：

- 主数字：`1540`
- 次级提示：`+240`

### 8.2 Drawer / 玩家详情

详细信息在 drawer 或玩家详情里展示：

- 初始带入
- 补码总额
- 总带入
- 当前筹码
- 本局净盈亏

### 8.3 结算窗口的短时反馈

在结算窗口期间，席位上短暂高亮：

- 绿色：本手净赢
- 红色：本手净输

## 9. 必须先修正的产品规则

### 9.1 开局权限

硬规则：

- 默认只有房主可开局
- 房主即使处于观战态，也可以开局，但不会被卷入本手
- 非房主绝不能代开
- 前端按钮只负责显示，不承担权限判断

### 9.2 入座与观战语义

以下状态必须有唯一文案、唯一操作项、唯一表现：

- 观战中
- 已入座，下一手加入
- 本手已弃牌
- 筹码不足，等待补码
- 房间异常，等待修复或重置

### 9.3 补码语义

建议统一为：

- 本手仍有参与资格：不能补码
- 本手已弃牌 / 已出局 / 本手已结束：可申请补码
- 前端文案必须与服务端真实规则一致

### 9.4 脏房处理

对例如以下异常状态：

- `gameStarted = true`
- 无有效当前行动人
- 无法推进当前手牌

服务端必须进入 `recovery_required` 或执行明确修复，而不是静默无效。

结果只能是：

- 自动修复到可继续状态
- 明确告知房间损坏，需要重置

## 10. UX 批判与优化原则

基于真实联机回归，以下不是视觉问题，而是必须进设计的体验问题：

- 结果弹窗曾真实阻断牌局节奏，因此必须移除
- 旧日志会重复、跨手、无法解释边池，因此必须由 `HandRecord` 替代
- `waiting` 文案和真实状态脱节，必须拆语义
- reconnect 后必须恢复到正确身份和正确阶段
- `auto fold / timeout check / disconnect fold` 必须在记录中显式标注
- 无边池场景绝不能生成边池展示

动画设计原则：

- 动画用于解释钱的流向，不替代信息
- 不遮挡关键操作区
- 不拖慢每手节奏
- 必须能解释平分池与多边池

## 11. 实施顺序

### Phase A：规则与状态机收正

目标：

- 拆分 `RoomState / HandState / SeatState / PlayerLedger`
- 修复开局权限
- 修复 `waiting` 语义污染
- 修复 mid-hand 抢座与 `0` 筹码死座位
- 收口补码、入座、观战规则
- 建立房间异常修复路径

### Phase B：重做牌局记录与结算窗口

目标：

- 引入 `HandRecord`
- 下线结算弹窗
- 新增 `settling` 窗口
- 接入 reveal 流程
- 让 drawer 成为 Hand History

### Phase C：重做桌面信息层

目标：

- 席位显示当前筹码与本局净盈亏
- 桌面高亮行动人、底池、下注、状态标签
- 结算时高亮赢家并执行筹码归集
- 清理错误或模糊文案

### Phase D：布局与视觉语言重设计

仅在 A/B/C 稳定后再做：

- 桌面与侧边信息区重排
- 桌面端与移动端布局差异
- 动画语言
- 字体、色彩、空间层级

## 12. 验收标准

必须通过以下真实联机回归：

1. 双人闭环
- 建房
- 入房
- 开局
- all-in / call
- 结算窗口
- 自动下一手

2. 三人边池
- 不等筹码
- 主池/边池拆分
- 平分池
- drawer 解释正确

3. 观战 / 中途入座
- 满座观战
- mid-hand 抢空座不再污染
- `waiting next hand` 文案与状态一致

4. 断线重连
- 当前行动人断线自动处理
- reconnect 恢复正确身份
- reconnect 到 `settling` 时恢复 reveal 权限和倒计时

5. 补码
- 合法时机允许补码
- 非法时机明确拒绝
- 前端入口与服务端规则一致
- `0` 筹码玩家可恢复但不污染房间

6. 六人连续多手 smoke
- 连续多手
- 庄位轮转正常
- 总筹码守恒
- 不出现错误 side pot
- 不出现跨手脏弹层或脏文案

## 13. 完成定义

只有同时满足以下条件，才认为本轮重构完成：

- 规则可信
- 状态语义可信
- 记录可信
- 结算不打断
- 出错时可解释、可恢复、可清理
